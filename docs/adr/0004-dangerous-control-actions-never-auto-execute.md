# Dangerous control actions are never auto-executed

The actuator (PR-Control-4) executes only the safe action whitelist — `increase_interval`, `pause`, `require_review` — all reversible and idempotent. When the feedback law suggests `force_cursor_rescan` or `switch_write_strategy`, the actuator does not execute the suggested action; it downgrades to executing `require_review` (flag the source for a human), while the original suggestion stays visible in advisory chips and the evidence ledger. Rationale: those two actions mutate data integrity (re-ingestion replay, write-path changes) — a class of error that anti-oscillation guards (cooldown, rate caps, TTL) cannot protect against; making them safe would require a dry-run/shadow-verification layer that roughly doubles the scope with no real-world evidence yet to justify it. Automating them later requires revisiting this ADR, not just extending the whitelist.
